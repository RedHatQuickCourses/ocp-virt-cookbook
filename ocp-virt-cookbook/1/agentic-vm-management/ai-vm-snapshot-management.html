VM Snapshot Management with Agentic Workflows

Versions tested:

OpenShift 4.22

This tutorial demonstrates how to manage the full virtual machine snapshot lifecycle using natural language commands through four agent skills: vm-snapshot-create, vm-snapshot-list, vm-snapshot-restore, and vm-snapshot-delete. You will walk through a realistic scenario of creating a pre-upgrade snapshot, listing available snapshots, restoring after a failed upgrade, verifying the restored VM, and cleaning up old snapshots.

Prerequisites

  • Completed the Getting Started with Agentic VM Management tutorial (skill pack installed, MCP server configured, cluster connectivity verified)

  • OpenShift cluster with OpenShift Virtualization installed

  • KUBECONFIG environment variable set and pointing to a valid kubeconfig file

  • At least one VM to manage (see Creating Virtual Machines with Natural Language to create one)

  • Storage backend with CSI snapshot support (a VolumeSnapshotClass must exist for your storage provisioner)

  • RBAC permissions to create, list, and delete VirtualMachineSnapshot and VirtualMachineRestore resources

Create the namespace used in this tutorial if it does not already exist:

oc create namespace my-vms

If you do not have a VM to manage, create one using the vm-create skill by typing in your IDE chat:

Create a Fedora VM named database-01 in namespace my-vms

Verify the VM exists before continuing:

oc get vm -n my-vms

Expected output:

NAME          AGE   STATUS    READY
database-01   5m    Running   True

Verify that your storage backend supports snapshots:

oc get volumesnapshotclass

Expected output (your driver and class name may differ):

NAME              DRIVER       DELETIONPOLICY   AGE
lvms-snapclass    topolvm.io   Delete           17d

If no VolumeSnapshotClass is returned, contact your cluster administrator to configure one before proceeding.

How It Works

The four snapshot skills cover the complete snapshot lifecycle. Each skill enforces human-in-the-loop confirmation before any operation that modifies or destroys data.

vm-snapshot-create Skill

The vm-snapshot-create skill follows a multi-step workflow:

  1. Verify VM and analyze storage — The agent confirms the VM exists, checks for hot-plugged volumes (which block snapshots), verifies the storage class supports CSI snapshots, and locates the matching VolumeSnapshotClass.

  2. Check VM state and guest agent — The agent determines whether the VM is running or stopped. If the storage backend supports online snapshots, the agent reports whether the QEMU guest agent is connected. A connected guest agent enables application-consistent snapshots through filesystem freeze/thaw. Without it, snapshots are crash-consistent.

  3. Estimate storage and present configuration — The agent estimates storage consumption, displays the exact YAML that will be applied, and waits for your explicit confirmation.

  4. Create and monitor — After confirmation, the agent creates the VirtualMachineSnapshot resource and monitors its status until it reaches Succeeded or Failed.

vm-snapshot-list Skill

The vm-snapshot-list skill is a read-only operation that requires no confirmation. It lists all snapshots in a namespace, optionally filtered by VM name, and displays each snapshot’s status, age, and readiness for restore.

vm-snapshot-restore Skill

The vm-snapshot-restore skill enforces the strictest safety model of any snapshot skill:

  1. Verify VM is stopped — Restoring a running VM would cause data corruption. If the VM is running, the agent offers to stop it first or cancel.

  2. Verify snapshot is ready — The agent confirms the snapshot exists and has readyToUse: true.

  3. Typed snapshot name confirmation — You must type the exact snapshot name to confirm. A mismatch cancels the operation.

  4. Final yes/cancel confirmation — After typed verification passes, the agent asks one more time before executing.

  5. Execute and monitor — The agent creates a VirtualMachineRestore resource and monitors progress until the restore completes.

vm-snapshot-delete Skill

The vm-snapshot-delete skill permanently removes a snapshot and frees its storage:

  1. Verify snapshot exists — The agent retrieves the snapshot details and lists other snapshots for the same VM.

  2. Warn if last snapshot — If this is the only snapshot for the VM, the agent displays a warning that no recovery points will remain.

  3. Typed snapshot name confirmation — You must type the exact snapshot name to confirm deletion.

  4. Delete and report — The agent deletes the snapshot and reports the result with remaining snapshots.

Snapshot Consistency Levels

The consistency of a snapshot depends on whether the VM is running and whether the QEMU guest agent is active:

VM State Guest Agent Consistency Level

Stopped

N/A

Offline snapshot — fully consistent, equivalent to clean shutdown

Running

Connected

Application-consistent — guest agent freezes filesystems during snapshot

Running

Not connected

Crash-consistent — equivalent to pulling the power cord; usable but may require filesystem recovery on restore

For production VMs, install the QEMU guest agent before taking snapshots to ensure application-consistent captures. On Linux VMs:

sudo dnf install qemu-guest-agent
sudo systemctl enable --now qemu-guest-agent

For Windows VMs, install the VirtIO drivers package which includes the guest agent and supports VSS (Volume Shadow Copy Service) for application-consistent snapshots of Windows workloads.

Example Scenario 1: Creating a Pre-Upgrade Snapshot

This scenario demonstrates creating a snapshot before upgrading application software on a VM. The snapshot serves as a rollback point if the upgrade fails.

  1. In your IDE chat, type:

    Create a snapshot of database-01 before I upgrade it

  2. The agent asks for the namespace:

    Which namespace is database-01 in?

  3. You provide the namespace:

    my-vms

  4. The agent verifies the VM, analyzes the storage backend, checks the guest agent status, and presents the configuration for review:

    ## VM Snapshot - Review Configuration

**VM to Snapshot:**
- **Name**: `database-01`
- **Namespace**: `my-vms`
- **Status**: Running
- **Storage**: 30Gi

**Storage Backend Analysis:**
- **Storage Class**: `lvms-vg1`
- **CSI Driver**: `topolvm.io`
- **Online Snapshots**: Supported
- **VolumeSnapshotClass**: `lvms-snapclass`

**Guest Agent Status:**
- **QEMU Guest Agent**: Connected and running
- **Filesystem Freeze/Thaw**: Will be performed during snapshot
- **Expected Consistency**: Application-consistent (with freeze)

**Snapshot Details:**
- **Snapshot Name**: `database-01-snapshot-20260714-100000` (auto-generated)
- **Type**: Online snapshot
- **Estimated Storage**: ~30Gi (initial snapshot)

**YAML to be applied:**

    apiVersion: snapshot.kubevirt.io/v1beta1
    kind: VirtualMachineSnapshot
    metadata:
      name: database-01-snapshot-20260714-100000
      namespace: my-vms
    spec:
      source:
        apiGroup: kubevirt.io
        kind: VirtualMachine
        name: database-01

**Proceed with snapshot creation? (yes/no)**

  5. You review the configuration and confirm:

    yes

  6. The agent creates the snapshot and monitors its progress:

Snapshot creation in progress...
  Status: InProgress

End-to-End Scenario: Pre-Upgrade Backup and Recovery

This section summarizes the complete workflow you practiced in the scenarios above. Use this as a reference checklist when performing upgrades on production VMs.

  1. Create a pre-upgrade snapshot — Before starting the upgrade, create a snapshot as a rollback point.

    Create a snapshot of database-01 in namespace my-vms

  2. Perform the upgrade — Carry out your upgrade procedure (install packages, update configurations, run migrations).

  3. Verify the upgrade — Test the application to confirm the upgrade succeeded.

  4. If the upgrade succeeded — Delete the pre-upgrade snapshot to free storage.

    Delete snapshot database-01-snapshot-20260714-100000 in namespace my-vms

  5. If the upgrade failed — Restore the VM from the pre-upgrade snapshot.

    Restore database-01 from snapshot database-01-snapshot-20260714-100000

  6. Start the restored VM — After restore completes, start the VM and verify it is healthy.

    Start VM database-01 in namespace my-vms

  7. Clean up old snapshots — After verifying the restored VM is healthy, remove any snapshots you no longer need.

    List snapshots for VM database-01 in namespace my-vms

Hot-Plugged Volume Restrictions

VMs with hot-plugged volumes (volumes attached after VM creation without restarting) cannot be snapshotted. If you attempt to create a snapshot for a VM with hot-plugged volumes, the agent reports the error and suggests resolution steps:

  1. Stop the VM to detach hot-plugged volumes.

  2. Persist the volumes in the VM spec so they are no longer considered hot-plugged.

  3. Restart the VM.

  4. Retry the snapshot operation.

Human-in-the-Loop Confirmation

All four snapshot skills enforce a safety model that prevents unintended data changes:

Read-only operations (vm-snapshot-list):

  • No confirmation required. The agent lists snapshots and reports results immediately.

Snapshot creation (vm-snapshot-create):

  • The agent analyzes the storage backend and guest agent status before presenting the configuration.

  • The exact YAML to be applied is displayed for your review.

  • You must explicitly confirm with "yes" before the snapshot is created.

Snapshot restore (vm-snapshot-restore):

  • The VM must be stopped before restore. If running, the agent offers to stop it.

  • You must type the exact snapshot name (case-sensitive) to confirm.

  • After typed verification, you must confirm with "yes" one more time.

  • Any mismatch or non-"yes" response cancels the operation.

Snapshot deletion (vm-snapshot-delete):

  • The agent shows the snapshot details and lists other available snapshots for the same VM.

  • If this is the last snapshot for the VM, a warning is displayed.

  • You must type the exact snapshot name (case-sensitive) to confirm.

  • A mismatch cancels the operation.

Common Issues

VolumeSnapshotClass Not Found

If the agent reports that no VolumeSnapshotClass exists for your storage provisioner, your storage backend may not be configured for snapshots. Check with your cluster administrator and verify with:

oc get volumesnapshotclass

Crash-Consistent Snapshot Warning

If the agent reports a crash-consistent snapshot instead of application-consistent, the QEMU guest agent is not running inside the VM. Install and enable it:

sudo dnf install qemu-guest-agent
sudo systemctl enable --now qemu-guest-agent

Verify the agent is connected:

oc get vmi database-01 -n my-vms -o jsonpath='{.status.conditions[?(@.type=="AgentConnected")].status}'
True

Restore Stuck in Progress

If a restore operation remains in InProgress status for more than 10 minutes, check the VirtualMachineRestore resource for error conditions:

oc get virtualmachinerestore -n my-vms -o yaml

Common causes include insufficient storage quota, CSI driver issues, or a corrupted snapshot. Contact your cluster administrator if the issue persists.

Cleanup

To remove snapshots created during this tutorial:

oc delete virtualmachinesnapshot --all -n my-vms

To remove the VM:

oc delete vm database-01 -n my-vms

To remove the namespace:

oc delete namespace my-vms

Summary

  • The vm-snapshot-create skill creates point-in-time snapshots with full storage backend validation, guest agent detection, and human confirmation before execution.

  • The vm-snapshot-list skill is a read-only operation that displays snapshots with their status, age, and readiness without requiring confirmation.

  • The vm-snapshot-restore skill enforces the strictest safety model: VM must be stopped, typed snapshot name confirmation, and a final yes/cancel gate before restoring.

  • The vm-snapshot-delete skill requires typed name confirmation and warns when deleting the last snapshot for a VM.

  • Application-consistent snapshots require the QEMU guest agent (Linux) or VirtIO drivers with VSS support (Windows) to be installed and running.

  • The end-to-end pre-upgrade workflow is: create snapshot, perform upgrade, verify, and either clean up (on success) or restore (on failure).
