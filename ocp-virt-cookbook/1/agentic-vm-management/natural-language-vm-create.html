Creating Virtual Machines with Natural Language

Versions tested:

OpenShift 4.22

This tutorial demonstrates how to create virtual machines using natural language commands through the vm-create agent skill. The agent interprets human-readable descriptions of VM requirements, gathers any missing parameters through interactive questions, and translates them into complete KubeVirt VirtualMachine resources with proper instance types, OS selection, storage configuration, and scheduling.

Prerequisites

  • Completed the Getting Started with Agentic VM Management tutorial (skill pack installed, MCP server configured, cluster connectivity verified)

  • OpenShift cluster with OpenShift Virtualization installed

  • KUBECONFIG environment variable set and pointing to a valid kubeconfig file

  • A namespace where you have permissions to create VirtualMachine resources

Create the namespaces used in this tutorial if they do not already exist:

oc create namespace my-vms
oc create namespace development

How It Works

The vm-create skill follows a six-step workflow:

  1. Gather requirements — The agent collects VM parameters through interactive questions. You can provide everything in a single prompt (e.g., "Create a Fedora VM named web-01 in namespace dev") or let the agent ask for missing details one at a time. Parameters you do not specify use sensible defaults.

  2. Check for conflicts — The agent verifies that no VM with the same name already exists in the target namespace. If a conflict is found, you are given the option to choose a different name, delete the existing VM, or cancel.

  3. Present configuration for confirmation — The agent displays a summary table showing every parameter that will be used. You can confirm, cancel, or modify individual values before anything is created.

  4. Create the VM — After your explicit confirmation, the agent calls the vm_create MCP tool to create the VirtualMachine resource.

  5. Verify and diagnose — The agent checks the VM status. If scheduling or storage errors occur (e.g., ErrorUnschedulable, ErrorDataVolumeNotReady), the agent automatically consults its troubleshooting knowledge base, diagnoses the root cause, and proposes a workaround for your approval.

  6. Report status — The agent reports whether the VM was created successfully and provides access instructions including default credentials and connection methods.

Example Scenario 1: Basic Fedora VM

This example creates a simple Fedora virtual machine with default settings.

  1. In your IDE chat, type:

    Create a Fedora VM in namespace my-vms

  2. The agent gathers cluster information (available StorageClasses, namespaces) and assigns defaults for parameters you did not specify. It then presents a configuration summary for your approval:

├──────────────┼───────────┼───────────────────────────────────┤
│ Namespace    │ my-vms    │ from request                      │
├──────────────┼───────────┼───────────────────────────────────┤
│ OS           │ fedora    │ Fedora Linux (latest)             │
├──────────────┼───────────┼───────────────────────────────────┤
│ Performance  │ u1        │ General Purpose                   │
├──────────────┼───────────┼───────────────────────────────────┤
│ Size         │ medium    │ 1 vCPU, 4Gi RAM                   │
├──────────────┼───────────┼───────────────────────────────────┤
│ Storage      │ 30Gi      │ root disk                         │
├──────────────┼───────────┼───────────────────────────────────┤
│ StorageClass │ lvms-vg1  │ cluster default (topolvm.io)      │
├──────────────┼───────────┼───────────────────────────────────┤

└──────────────┴───────────┴───────────────────────────────────┘

Confirm: yes / no / modify?

  3. You review the table and confirm:

    yes

  4. The agent creates the VM using the vm_create MCP tool. The underlying VirtualMachine resource it generates looks like this:

    apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-vm
  namespace: my-vms
spec:
  instancetype:
    kind: VirtualMachineClusterInstancetype
    name: u1.medium
  preference:
    kind: VirtualMachineClusterPreference
    name: fedora
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: fedora-vm-rootdisk
    spec:
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
      storage:
        resources:
          requests:
            storage: 30Gi
  template:
    spec:
      domain:
        devices: {}
        resources: {}
      terminationGracePeriodSeconds: 180
      volumes:
      - dataVolume:
          name: fedora-vm-rootdisk
        name: fedora-vm-rootdisk

  5. The agent monitors the DataVolume provisioning (cloning from the golden image takes approximately 2–5 minutes) and reports the result once the VM is ready:

    ✓ Virtual Machine Created Successfully

VM: fedora-vm (namespace: my-vms)
OS: Fedora | Size: medium (u1) | Storage: 30Gi | Status: Running

Accessing the VM
1. Serial console: virtctl console fedora-vm -n my-vms
2. VNC: OpenShift Console → Virtualization → VMs → fedora-vm → Console
3. SSH: Get IP from VMI, ssh fedora@<ip>

Default credentials: fedora

After creation you can verify the VM with:

oc get vm fedora-vm -n my-vms

Expected output:

fedora-vm   7m    Running   True

Example Scenario 3: Error Diagnosis on Multi-Node Clusters

This scenario demonstrates the agent’s automatic error diagnosis capability. It requires a multi-node cluster where worker nodes have taints configured (e.g., virtualization=true:NoSchedule). On a single-node cluster without taints, VMs schedule successfully and this error will not occur.

This example shows what happens when a VM creation triggers a scheduling error and how the agent diagnoses and resolves it.

  1. In your IDE chat, type:

    Create a Fedora VM named fedora-db with 4 vCPUs and 16GB RAM in the production namespace, memory-optimized, with 100GB storage, and autostart

  2. The agent maps the request to an xlarge size with m1 (memory-optimized) performance profile and presents the confirmation table:

├──────────────┼──────────────┼────────────────────────────────────┤
│ OS           │ fedora       │ Fedora Linux (latest)              │
├──────────────┼──────────────┼────────────────────────────────────┤
│ Performance  │ m1           │ Memory Optimized                   │
├──────────────┼──────────────┼────────────────────────────────────┤
│ Size         │ xlarge       │ 4 vCPU, 32Gi RAM                   │
├──────────────┼──────────────┼────────────────────────────────────┤
│ Storage      │ 100Gi        │ root disk                          │
├──────────────┼──────────────┼────────────────────────────────────┤

  3. After confirmation, the agent creates the VM but detects a scheduling error:

  2) Manual fix (I'll show you the oc patch command)
  3) Cancel (delete the VM)

  5. The agent patches the VM spec to add tolerations, restarts it, and verifies:

2. VNC: OpenShift Console → Virtualization → VMs → fedora-db → Console
3. SSH: Get IP from VMI, ssh fedora@<ip>

Instance Type Resolution

The agent uses two dimensions to select the appropriate VirtualMachineClusterInstancetype: performance profile and size.

Performance Profiles

Performance profiles determine the workload optimization of the instance type:

Profile Name Best For

u1

General Purpose (default)

Balanced CPU and memory — suitable for most workloads

m1

Memory Optimized

Memory-intensive workloads like databases or caches

o1

Overcommitted

Development and testing — shares physical resources
Additional profiles may be available depending on your cluster configuration. Run oc get VirtualMachineClusterInstancetype to see all installed instance types on your cluster.

The agent automatically:

  • Queries the cluster for available StorageClasses

  • Identifies the default StorageClass (annotated with storageclass.kubernetes.io/is-default-class)

  • Analyzes the StorageClass capabilities (volume binding mode, CSI provisioner)

  • Creates a DataVolume-backed PersistentVolumeClaim with the appropriate size

Network Setup

By default, the agent creates VMs connected to the cluster’s default pod network, which provides basic connectivity for most use cases.

If your cluster has secondary network attachment definitions configured, you can request additional network interfaces in your natural language prompt (e.g., "create a VM with a secondary network on vlan-100"). The agent passes the network configuration to the vm_create tool.

Human-in-the-Loop Confirmation

Every VM creation includes an explicit confirmation step. The agent will never create cluster resources without your approval:

  1. The agent gathers all parameters — from your prompt, interactive questions, and cluster defaults.

  2. A configuration table is displayed showing every value that will be used.

  3. You choose one of:

    • yes — proceed with creation

    • no — cancel the operation

    • modify — change specific parameters (e.g., "modify: change storage to 50Gi")

  4. After modification, the updated table is shown and you confirm again.

This safety model prevents unintended resource consumption and gives you full visibility into what will be created.

Automatic Error Diagnosis

If the VM enters an error state after creation, the agent automatically diagnoses the issue using its built-in troubleshooting knowledge base.

Scheduling Errors (ErrorUnschedulable)

The three most common causes are:

  1. Node taints (most common) — Worker nodes have taints (e.g., virtualization=true:NoSchedule) that the VM does not tolerate. The agent can add the required tolerations to the VM spec and restart it.

  2. Insufficient resources — The cluster does not have enough CPU, memory, or storage on any node for the requested VM size. The agent suggests reducing the VM size or checking node resource availability.

  3. Node selector mismatch — The VM requires specific node labels that do not exist in the cluster. The agent can remove the selector or suggest adding labels to nodes.

Storage Errors (ErrorDataVolumeNotReady)

Common storage issues include:

  • DataVolume still provisioning — Large disk imports can take several minutes. The agent monitors progress and reports when provisioning completes.

  • StorageClass not found — The requested StorageClass does not exist. The agent lists available options and offers to recreate the VM with a valid StorageClass.

  • Insufficient storage quota — The namespace quota does not allow the requested disk size. The agent reports current quota usage and suggests alternatives.

In all cases, the agent presents a clear diagnosis with recommended solutions and waits for your decision before taking any corrective action.

Accessing Your VM

After the VM is created and running, you can connect using several methods:

Method Command

Serial console

virtctl console <vm-name> -n <namespace>

VNC console

OpenShift Console → Virtualization → VirtualMachines → select VM → Console tab

SSH

Get the VM IP from the VirtualMachineInstance, then ssh <user>@<ip>

Port forwarding

virtctl port-forward vmi/<vm-name> -n <namespace> 8080:80

To find the VM’s IP address:

oc get vmi fedora-vm -n my-vms -o jsonpath='{.status.interfaces[0].ipAddress}'
10.128.0.102

Default Credentials

Each OS image ships with a default user account. Console access is available immediately; for SSH, you need to set a password or inject an SSH key:

# Connect via serial console and set a password
virtctl console fedora-vm -n my-vms
# Once logged in:
sudo passwd fedora

Cleanup

To remove a VM and its associated storage:

oc delete vm <vm-name> -n <namespace>

Deleting the VirtualMachine resource also deletes the associated DataVolume and PersistentVolumeClaim. To remove all VMs created in this tutorial:

oc delete vm fedora-vm -n my-vms
oc delete vm fedora-dev -n development

You can also use natural language: type "Delete VM fedora-vm in namespace my-vms" and the agent will use the vm-delete skill, which includes a name-confirmation safety check before deletion.
