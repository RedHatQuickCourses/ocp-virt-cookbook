Creating Windows Golden Images

Overview

This tutorial walks you through creating a Windows Server 2025 golden image on OpenShift Virtualization using the windows-efi-installer Tekton pipeline. The pipeline automates the full workflow: downloading the ISO, modifying it for unattended boot, running a silent installation with VirtIO drivers, and generalizing the image with sysprep. Once complete, you can deploy new Windows VMs from this golden image, with per-clone customization handled through a sysprep unattend.xml file.

The pipeline approach is necessary for Windows because Windows lacks an equivalent to Linux’s cloud-init. Instead, Windows uses autounattend.xml for unattended installation and sysprep for generalization. Automating these steps by hand is error-prone, so the pipeline handles it end to end.

What You Will Learn

  • How Windows golden images differ from Linux golden images

  • How to configure an autounattend.xml for unattended Windows Server 2025 installation

  • How to run the windows-efi-installer Tekton pipeline to build a golden image

  • How to deploy VMs from the golden image with per-clone sysprep customization

  • How to adapt the process for other Windows versions (10, 11, Server 2022)

How Windows Golden Images Differ from Linux

Linux and Windows use different mechanisms at each stage of the golden image workflow:

Stage Linux Windows

Unattended install

Kickstart or preseed

autounattend.xml (Windows Setup answer file)

Post-install customization

cloud-init on first boot

unattend.xml via sysprep on first boot (OOBE pass)

Generalization

Remove /etc/machine-id, run cloud-init clean

Run sysprep /generalize /oobe /shutdown

Per-clone identity

cloud-init userData (hostname, SSH keys, packages)

Sysprep unattend.xml volume (hostname, admin password, RDP)

Driver injection

Drivers typically included in kernel

VirtIO drivers must be loaded during installation

The windows-efi-installer pipeline automates all of these steps. It downloads the ISO, patches the bootloader for unattended boot, creates a blank disk, boots an installer VM with the autounattend.xml ConfigMap and the VirtIO driver container disk, waits for installation and sysprep to finish, and publishes the resulting disk as a DataSource.

Prerequisites

  • OpenShift 4.18+ with OpenShift Virtualization operator installed

  • OpenShift Pipelines operator installed (provides Tekton)

  • CLI tools: oc and kubectl

  • A Windows Server 2025 evaluation ISO download URL from the Microsoft Evaluation Center

  • A storage class with at least 60 GiB available for the root disk

  • Network access from the cluster to go.microsoft.com (or an internal ISO mirror)

  • Familiarity with the Linux golden image workflow

Step 1: Prepare the Namespace and Service Account

The pipeline runs in the default namespace. The modify-windows-iso-file pipeline task requires specific user and group IDs to write to its workspace volume. Grant the anyuid Security Context Constraint (SCC) to the pipeline service account:

oc adm policy add-scc-to-user anyuid -z pipeline -n default

This allows the task pod to run with runAsUser: 107 and fsGroup: 107, which the ISO modification tool requires.

Step 2: Obtain the Windows ISO URL

  1. Visit the Microsoft Evaluation Center for Windows Server 2025.

  2. Sign in with a Microsoft account (free registration).

  3. Select 64-bit edition, language English (United States), and fill in the registration form.

  4. On the download page, right-click the Download button and copy the link address. Do not click the button directly.

  5. Save this URL. You will use it in the PipelineRun manifest.

The download URL is temporary and expires after approximately 24 hours. If the pipeline fails to download the ISO, generate a new URL from the Evaluation Center.
Evaluation ISOs have a 180-day trial limit. For production use, use a licensed ISO from your volume licensing portal.

For other Windows versions, use these download pages:

  • Windows 10 Enterprise

  • Windows 11 Enterprise

  • Windows Server 2022

Step 3: Create the Autounattend ConfigMap

The autounattend.xml file tells Windows Setup how to install unattended, and the post-install.ps1 script runs after installation to install drivers and prepare the image for sysprep. Both files are stored in a single ConfigMap with two keys.

    Start-Process msiexec -Wait -ArgumentList "/i E:\guest-agent\qemu-ga-x86_64.msi /qn /passive /norestart"

    mv C:\Windows\Panther\unattend.xml C:\Windows\Panther\unattend.install.xml

    # Eject CD to prevent autounattend.xml on CD from being picked up by sysprep
    (New-Object -COMObject Shell.Application).NameSpace(17).ParseName("F:").InvokeVerb("Eject")

Apply the ConfigMap:

oc apply -f configmap-windows2k25-autounattend.yaml

What the autounattend.xml does

The file contains four Windows Setup passes:

  1. windowsPE — Runs during the Windows Preinstallation Environment before files are copied to disk. It loads VirtIO storage (viostor) and network (NetKVM) drivers so the installer can see the VirtIO SCSI disk and NIC. It also partitions the disk with GPT layout (EFI, MSR, and primary partitions), selects the Windows Server 2025 Datacenter (Desktop Experience) edition, and accepts the EULA.

  2. specialize — Runs after files are copied to disk. Sets regional and language preferences to en-US.

  3. oobeSystem — Normally this pass runs the Out-Of-Box Experience (OOBE) wizard. Here it is configured to reseal into Audit mode instead, which skips OOBE and boots directly to the desktop as the built-in Administrator so the post-install script can run.

  4. auditUser — Runs in Audit mode. It executes post-install.ps1, then calls sysprep /generalize /oobe /shutdown to strip machine-specific identifiers and shut down the VM. The resulting disk is a clean, generalized golden image.

What the post-install.ps1 does

The PowerShell script runs as SYSTEM in Audit mode and performs four tasks:

  1. Installs the VirtIO guest driver package (virtio-win-gt-x64.msi) from the VirtIO container disk mounted at E:\.

  2. Installs the QEMU guest agent (qemu-ga-x86_64.msi) so that OpenShift can communicate with the VM after deployment.

  3. Renames the cached C:\Windows\Panther\unattend.xml to unattend.install.xml. This prevents sysprep from picking up the install-time answer file when a cloned VM boots for the first time.

  4. Ejects the CD-ROM drive to prevent the autounattend.xml on the CD from being reused on subsequent boots.

Step 4: Run the Pipeline

Replace REPLACE_WITH_YOUR_ISO_URL in the following manifest with the URL you copied in Step 2.

Download pipelinerun-windows2k25.yaml

apiVersion: tekton.dev/v1
kind: PipelineRun
metadata:
  generateName: windows2k25-installer-run-
  namespace: default
spec:
  params:
    - name: winImageDownloadURL
      value: "REPLACE_WITH_YOUR_ISO_URL"
    - name: acceptEula
      value: "true"
    - name: preferenceName
      value: windows.2k25.virtio
    - name: autounattendConfigMapName
      value: windows2k25-autounattend
    - name: baseDvName
      value: win2k25
    - name: isoDVName
      value: win2k25
  pipelineRef:
    params:
      - name: catalog
        value: redhat-pipelines
      - name: type
        value: artifact
      - name: kind
        value: pipeline
      - name: name
        value: windows-efi-installer
      - name: version
        value: v4.21.0
    resolver: hub
  taskRunSpecs:
    - pipelineTaskName: modify-windows-iso-file
      podTemplate:
        securityContext:
          fsGroup: 107
          runAsUser: 107
  timeouts:
    pipeline: 1h0m0s

Create the PipelineRun. Use oc create instead of oc apply because the manifest uses generateName:

oc create -f pipelinerun-windows2k25.yaml

Monitor the pipeline progress:

oc get pipelinerun -n default -w

Press Ctrl+C to exit the watch once the PipelineRun shows Succeeded.

You can also monitor the DataVolumes created by the pipeline:

oc get dv -n default -w

What the pipeline does

The windows-efi-installer pipeline runs the following stages:

  1. Import autounattend ConfigMap — Reads the autounattend.xml and post-install.ps1 from the ConfigMap you created.

  2. Download Windows ISO — Downloads the ISO into a DataVolume.

  3. Modify ISO — Replaces the interactive EFI bootloader (efisys.bin) with a no-prompt bootloader (efisys_noprompt.bin) so the installer boots without waiting for "Press any key."

  4. Create blank root disk — Creates a 60 GiB DataVolume for the Windows installation target.

  5. Boot installer VM — Starts a VM with the modified ISO, the blank root disk, and the VirtIO driver container disk. The autounattend.xml drives the entire installation without user interaction.

  6. Wait for shutdown — The pipeline waits for the VM to shut down, which happens after sysprep generalize completes in the auditUser pass.

  7. Create DataSource — Creates a DataSource pointing to the installed root disk, making it available as a clone source.

  8. Clean up — Removes the installer VM and temporary DataVolumes (ISO, modified ISO).

The full pipeline takes 30 to 60 minutes depending on network speed (ISO download) and storage performance (disk cloning).

Step 6: Deploy a VM from the Golden Image

Each VM cloned from the golden image needs its own sysprep unattend.xml to configure machine-specific settings during the first boot OOBE pass. This file is stored in a ConfigMap and attached to the VM as a sysprep volume. KubeVirt mounts it as a floppy or CD and Windows Setup reads it automatically during OOBE.

Create the sysprep ConfigMap

The unattend.xml below sets the hostname to WIN-SERVER-01, sets the Administrator password, enables auto-logon, extends the OS partition to fill the disk, enables Remote Desktop, and skips all OOBE screens.

The password in this example is for demo purposes only. Use a strong, unique password in production environments.

Updating the Golden Image

To create an updated golden image:

  1. Generate a new ISO download URL from the Microsoft Evaluation Center (or use an updated licensed ISO).

  2. Modify the autounattend.xml if needed (for example, to add new software in the post-install script).

  3. Re-run the pipeline with a different baseDvName value, such as win2k25-v2.

  4. Update your VM manifests to reference the new DataSource name.

Use versioned DataSource names (win2k25-v1, win2k25-v2, win2k25-2025-07) so you can roll back to a previous version if needed. Keep at least one previous version available.

Windows 11

(default)

viostor/w11/amd64, NetKVM/w11/amd64

Troubleshooting

Problem Solution

Pipeline fails at modify-windows-iso-file with "file not found: efisys_noprompt.bin"

The ISO may not contain the no-prompt EFI bootloader. This file is present in standard Microsoft ISOs but may be missing from custom or slimmed-down ISOs. Use an unmodified ISO from the Microsoft Evaluation Center.

Permission denied when modifying ISO

Add the taskRunSpecs section to the PipelineRun with fsGroup: 107 and runAsUser: 107 as shown in Step 4. Also confirm the anyuid SCC is granted to the pipeline service account.

Pipeline times out

Increase timeouts.pipeline in the PipelineRun manifest. Slow network connections or storage backends can cause the pipeline to exceed the default 1-hour timeout. Check DataVolume progress with oc get dv -w.

Cloned VM boots to OOBE wizard instead of desktop

The sysprep ConfigMap key must be unattend.xml, not autounattend.xml. Windows only reads autounattend.xml from removable media during a fresh install. After sysprep, it reads unattend.xml. Also verify the ConfigMap name in the VM’s volumes.sysprep.configMap.name matches the ConfigMap you created.

Sysprep fails with "rearm count exceeded"

Evaluation ISOs have a limited number of sysprep rearm cycles. Rebuild the golden image from a fresh ISO rather than re-sysprepping an existing image. For production use, use a volume-licensed ISO that does not have rearm limits.

Cloned VM slow to respond after boot

Windows OOBE runs in the background for up to 5 minutes on first boot. Wait for the guest agent to report guestOSInfo before assuming the VM is unresponsive.

Guest agent not reporting OS info

Verify the QEMU guest agent was installed by the post-install.ps1 script. Connect to the VM console and check that the QEMU Guest Agent service is running in services.msc.

Cleanup

Remove all resources created in this tutorial:

oc delete vm win2k25-vm-1 -n default
oc delete configmap win2k25-sysprep-unattend -n default
oc delete configmap windows2k25-autounattend -n default
oc delete datasource win2k25 -n default
oc delete dv win2k25 -n default

To also remove the PipelineRun resources:

oc delete pipelinerun -l tekton.dev/pipeline=windows-efi-installer -n default

To revoke the SCC granted in Step 1:

oc adm policy remove-scc-from-user anyuid -z pipeline -n default
