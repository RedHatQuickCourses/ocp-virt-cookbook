= Storage Profiles

== Understanding Storage Profiles and Their Role
In OpenShift Virtualization, a Storage Profile (StorageProfile) is a custom resource managed by the Containerized Data Importer (CDI). Its primary role is to tell OpenShift Virtualization how to optimally request storage (Persistent Volume Claims or PVCs) for Virtual Machines based on the underlying Storage Class.

When you create a VM or clone a disk, OpenShift needs to know:

Should the disk be provisioned as a Filesystem or a raw Block device?

Can the disk be shared across nodes (ReadWriteMany) to support Live Migration, or is it restricted to a single node (ReadWriteOnce)?

Instead of forcing developers to specify these details manually every time they deploy a VM, the StorageProfile maps the optimal accessModes and volumeMode to a specific StorageClass.

== Default Storage Profile Configuration
You rarely need to create a StorageProfile from scratch. The CDI controller actively monitors your cluster. Whenever a new StorageClass is created (e.g., by your CSI driver like OpenShift Data Foundation/Ceph or a SAN plugin), CDI automatically generates a matching StorageProfile with the exact same name.

CDI attempts to auto-detect the optimal parameters by querying the CSI driver's capabilities. If it cannot determine the optimal settings, it falls back to empty defaults, which typically resolve to ReadWriteOnce and Filesystem.

== Storage Profile Parameters
To customize a storage profile, you define Claim Property Sets (claimPropertySets). These sets represent the preferred configurations when provisioning a PVC for a VM.

The two main parameters are:

volumeMode:

Block: Provides raw block storage to the VM. (Highly Recommended for VMs). It eliminates the overhead of a filesystem, offering significantly better I/O performance.

Filesystem: The PVC is formatted with a filesystem (like ext4 or xfs). The VM's disk is stored as an .img file inside this filesystem.

accessModes:

ReadWriteMany (RWX): The volume can be mounted as read-write by many nodes. (Required for Live Migration).

ReadWriteOnce (RWO): The volume can be mounted as read-write by a single node. The VM must be powered down to move to another node.

== StorageProfile Resource Definition & Customization
If your default StorageProfile is configured for RWO/Filesystem, but you know your backend storage (like Ceph RBD or an NFS share) supports RWX/Block, you should patch the auto-generated profile.

Example: A Custom StorageProfile YAML
Here is what a customized StorageProfile looks like. In this example, we configure a storage class named ocs-storagecluster-ceph-rbd to prefer Block and ReadWriteMany.

[source,yaml]
----
apiVersion: cdi.kubevirt.io/v1beta1
kind: StorageProfile
metadata:
  name: ocs-storagecluster-ceph-rbd
spec:
  claimPropertySets:
  - accessModes:
    - ReadWriteMany
    volumeMode: Block
  - accessModes:
    - ReadWriteOnce
    volumeMode: Filesystem
----
NOTE: The CDI controller reads the claimPropertySets list from top to bottom. The first set in the array is treated as the primary preference.

How to Patch an Existing Profile:
Instead of applying a new YAML, you typically patch the existing profile generated by CDI:

[source,bash,role=execute]
----
oc patch storageprofile <storage_class_name> --type='json' -p='[{"op": "replace", "path": "/spec/claimPropertySets", "value": [{"accessModes": ["ReadWriteMany"], "volumeMode": "Block"}]}]'
----
== Verification Commands and Expected Outputs
List all storage profiles in the cluster:

[source,bash,role=execute]
----
oc get storageprofile
Expected Output:

Plaintext

NAME                          AGE
hostpath-provisioner          10d
ocs-storagecluster-ceph-rbd   10d
ocs-storagecluster-cephfs     10d
Check the configured properties of a specific profile:
----
[source,bash,role=execute]
----
oc get storageprofile ocs-storagecluster-ceph-rbd -o yaml
Expected Output (Truncated):
----
[source,yaml]
----
status:
  claimPropertySets:
  - accessModes:
    - ReadWriteMany
    volumeMode: Block
  provisioner: openshift-storage.rbd.csi.ceph.com
  storageClass: ocs-storagecluster-ceph-rbd
----

== Common Troubleshooting Scenarios
Scenario A: Virtual Machine Live Migration Fails

Symptom: You attempt to live-migrate a VM, but it is rejected with a "Not migratable" condition or gets stuck in a Paused state.

Cause: The VM's disk was provisioned as ReadWriteOnce. Live migration requires the destination node to mount the disk before the source node detaches it, which requires ReadWriteMany.

Fix: Check oc get storageprofile <storage_class> -o yaml. Ensure that ReadWriteMany is the default in the claimPropertySets. Patch it if necessary, then recreate the VM's disk so it inherits the new RWX profile.

Scenario B: Cloning a VM Fails or Hangs

Symptom: When cloning a VM from a template, the DataVolume gets stuck, and events show "DataVolume.storage spec is missing accessMode and volumeMode, cannot get access mode from StorageProfile".

Cause: The StorageProfile for your destination StorageClass has empty or misconfigured claimPropertySets. The cloner doesn't know what access mode or volume mode to use.

Fix: Explicitly define the claimPropertySets in the StorageProfile using the oc patch command shown in Section 4.
