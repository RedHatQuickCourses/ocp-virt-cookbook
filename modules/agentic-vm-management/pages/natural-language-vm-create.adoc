= Creating Virtual Machines with Natural Language
:navtitle: Natural Language VM Creation

Versions tested:
----
OpenShift 4.22
----

This tutorial demonstrates how to create virtual machines using natural language commands through the `vm-create` agent skill. The agent interprets human-readable descriptions of VM requirements, gathers any missing parameters through interactive questions, and translates them into complete KubeVirt VirtualMachine resources with proper instance types, OS selection, storage configuration, and scheduling.

== Prerequisites

* Completed the xref:getting-started.adoc[Getting Started with Agentic VM Management] tutorial (skill pack installed, MCP server configured, cluster connectivity verified)
* OpenShift cluster with OpenShift Virtualization installed
* `KUBECONFIG` environment variable set and pointing to a valid kubeconfig file
* A namespace where you have permissions to create VirtualMachine resources

Create the namespaces used in this tutorial if they do not already exist:

[source,bash,role=execute]
----
oc create namespace my-vms
oc create namespace development
----

== How It Works

The `vm-create` skill follows a six-step workflow:

. **Gather requirements** — The agent collects VM parameters through interactive questions. You can provide everything in a single prompt (e.g., "Create a Fedora VM named web-01 in namespace dev") or let the agent ask for missing details one at a time. Parameters you do not specify use sensible defaults.

. **Check for conflicts** — The agent verifies that no VM with the same name already exists in the target namespace. If a conflict is found, you are given the option to choose a different name, delete the existing VM, or cancel.

. **Present configuration for confirmation** — The agent displays a summary table showing every parameter that will be used. You can confirm, cancel, or modify individual values before anything is created.

. **Create the VM** — After your explicit confirmation, the agent calls the `vm_create` MCP tool to create the VirtualMachine resource.

. **Verify and diagnose** — The agent checks the VM status. If scheduling or storage errors occur (e.g., `ErrorUnschedulable`, `ErrorDataVolumeNotReady`), the agent automatically consults its troubleshooting knowledge base, diagnoses the root cause, and proposes a workaround for your approval.

. **Report status** — The agent reports whether the VM was created successfully and provides access instructions including default credentials and connection methods.

== Example Scenario 1: Basic Fedora VM

This example creates a simple Fedora virtual machine with default settings.

. In your IDE chat, type:
+
[source,text]
----
Create a Fedora VM in namespace my-vms
----

. The agent gathers cluster information (available StorageClasses, namespaces) and assigns defaults for parameters you did not specify. It then presents a configuration summary for your approval:
+
----
Here's the full configuration before I proceed:

Virtual Machine Configuration

┌──────────────┬───────────┬───────────────────────────────────┐
│  Parameter   │   Value   │               Notes               │
├──────────────┼───────────┼───────────────────────────────────┤
│ VM Name      │ fedora-vm │ validated                         │
├──────────────┼───────────┼───────────────────────────────────┤
│ Namespace    │ my-vms    │ from request                      │
├──────────────┼───────────┼───────────────────────────────────┤
│ OS           │ fedora    │ Fedora Linux (latest)             │
├──────────────┼───────────┼───────────────────────────────────┤
│ Performance  │ u1        │ General Purpose                   │
├──────────────┼───────────┼───────────────────────────────────┤
│ Size         │ medium    │ 1 vCPU, 4Gi RAM                   │
├──────────────┼───────────┼───────────────────────────────────┤
│ Storage      │ 30Gi      │ root disk                         │
├──────────────┼───────────┼───────────────────────────────────┤
│ StorageClass │ lvms-vg1  │ cluster default (topolvm.io)      │
├──────────────┼───────────┼───────────────────────────────────┤
│ Autostart    │ no        │ start manually after creation     │
└──────────────┴───────────┴───────────────────────────────────┘

This will create a VirtualMachine and consume cluster resources.
Confirm: yes / no / modify?
----

. You review the table and confirm:
+
[source,text]
----
yes
----

. The agent creates the VM using the `vm_create` MCP tool. The underlying VirtualMachine resource it generates looks like this:
+
[source,yaml]
----
apiVersion: kubevirt.io/v1
kind: VirtualMachine
metadata:
  name: fedora-vm
  namespace: my-vms
spec:
  instancetype:
    kind: VirtualMachineClusterInstancetype
    name: u1.medium
  preference:
    kind: VirtualMachineClusterPreference
    name: fedora
  runStrategy: Always
  dataVolumeTemplates:
  - metadata:
      name: fedora-vm-rootdisk
    spec:
      sourceRef:
        kind: DataSource
        name: fedora
        namespace: openshift-virtualization-os-images
      storage:
        resources:
          requests:
            storage: 30Gi
  template:
    spec:
      domain:
        devices: {}
        resources: {}
      terminationGracePeriodSeconds: 180
      volumes:
      - dataVolume:
          name: fedora-vm-rootdisk
        name: fedora-vm-rootdisk
----

. The agent monitors the DataVolume provisioning (cloning from the golden image takes approximately 2–5 minutes) and reports the result once the VM is ready:
+
----
✓ Virtual Machine Created Successfully

VM: fedora-vm (namespace: my-vms)
OS: Fedora | Size: medium (u1) | Storage: 30Gi | Status: Running

Accessing the VM
1. Serial console: virtctl console fedora-vm -n my-vms
2. VNC: OpenShift Console → Virtualization → VMs → fedora-vm → Console
3. SSH: Get IP from VMI, ssh fedora@<ip>

Default credentials: fedora
----

After creation you can verify the VM with:

[source,bash,role=execute]
----
oc get vm fedora-vm -n my-vms
----

Expected output:

----
NAME        AGE   STATUS    READY
fedora-vm   7m    Running   True
----

You can also inspect the DataVolume and PVC created for the root disk:

[source,bash,role=execute]
----
oc get datavolume -n my-vms
----

----
NAME                 PHASE       PROGRESS   RESTARTS   AGE
fedora-vm-rootdisk   Succeeded   100.0%                7m
----

[source,bash,role=execute]
----
oc get pvc -n my-vms
----

----
NAME                 STATUS   VOLUME                                     CAPACITY   ACCESS MODES   STORAGECLASS
fedora-vm-rootdisk   Bound    pvc-ebfcfd3d-f865-4186-b306-08dfc1d60272   30Gi       RWO            lvms-vg1
----

== Example Scenario 3: Error Diagnosis on Multi-Node Clusters

NOTE: This scenario demonstrates the agent's automatic error diagnosis capability. It requires a multi-node cluster where worker nodes have taints configured (e.g., `virtualization=true:NoSchedule`). On a single-node cluster without taints, VMs schedule successfully and this error will not occur.

This example shows what happens when a VM creation triggers a scheduling error and how the agent diagnoses and resolves it.

. In your IDE chat, type:
+
[source,text]
----
Create a Fedora VM named fedora-db with 4 vCPUs and 16GB RAM in the production namespace, memory-optimized, with 100GB storage, and autostart
----

. The agent maps the request to an `xlarge` size with `m1` (memory-optimized) performance profile and presents the confirmation table:
+
----

│  Parameter   │    Value     │               Notes                │
├──────────────┼──────────────┼────────────────────────────────────┤
│ VM Name      │ fedora-db    │ validated                          │
├──────────────┼──────────────┼────────────────────────────────────┤
│ Namespace    │ production   │ from request                       │
├──────────────┼──────────────┼────────────────────────────────────┤
│ OS           │ fedora       │ Fedora Linux (latest)              │
├──────────────┼──────────────┼────────────────────────────────────┤
│ Performance  │ m1           │ Memory Optimized                   │
├──────────────┼──────────────┼────────────────────────────────────┤
│ Size         │ xlarge       │ 4 vCPU, 32Gi RAM                   │
├──────────────┼──────────────┼────────────────────────────────────┤
│ Storage      │ 100Gi        │ root disk                          │
├──────────────┼──────────────┼────────────────────────────────────┤
│ StorageClass │ lvms-vg1     │ cluster default (topolvm.io)       │
├──────────────┼──────────────┼────────────────────────────────────┤

└──────────────┴──────────────┴────────────────────────────────────┘

. After confirmation, the agent creates the VM but detects a scheduling error:

⚠️ VM Scheduling Issue Detected

            that the pod didn't tolerate

          spec using resources_create_or_update, then restart the VM
  Impact: The VM will be able to schedule on nodes with the
          virtualization=true taint

  1) Apply workaround (add toleration and restart)
  2) Manual fix (I'll show you the oc patch command)
  3) Cancel (delete the VM)
----

. You confirm the workaround:
+
[source,text]

----

. The agent patches the VM spec to add tolerations, restarts it, and verifies:
+

Action: Added toleration for virtualization=true:NoSchedule, restarted VM
Status: Running

VM: fedora-db (namespace: production)
OS: Fedora | Size: xlarge (m1 - Memory) | Storage: 100Gi | Status: Running

Accessing the VM
1. Serial console: virtctl console fedora-db -n production
2. VNC: OpenShift Console → Virtualization → VMs → fedora-db → Console
3. SSH: Get IP from VMI, ssh fedora@<ip>

== Instance Type Resolution

The agent uses two dimensions to select the appropriate `VirtualMachineClusterInstancetype`: performance profile and size.

=== Performance Profiles

Performance profiles determine the workload optimization of the instance type:

[cols="1,2,3",options="header"]
|===
| Profile | Name | Best For
| `u1` | General Purpose (default) | Balanced CPU and memory — suitable for most workloads
| `m1` | Memory Optimized | Memory-intensive workloads like databases or caches
| `o1` | Overcommitted | Development and testing — shares physical resources
|===

NOTE: Additional profiles may be available depending on your cluster configuration. Run `oc get VirtualMachineClusterInstancetype` to see all installed instance types on your cluster.

=== Size Mappings

The following table shows the actual CPU and memory allocation for each `u1` (General Purpose) size as installed on OpenShift 4.22:

[cols="2,1,1",options="header"]
|===
| Size | vCPUs | RAM
| nano | 1 | 512Mi
| micro | 1 | 1Gi
| small | 1 | 2Gi
| medium (default) | 1 | 4Gi
| large | 2 | 8Gi
| xlarge | 4 | 16Gi
| 2xlarge | 8 | 32Gi
| 4xlarge | 16 | 64Gi
| 8xlarge | 32 | 128Gi
|===

Memory-optimized (`m1`) sizes allocate more RAM per vCPU. For example, `m1.large` provides 2 vCPU and 16Gi RAM, while `u1.large` provides 2 vCPU and 8Gi RAM.

The agent maps explicit resource requests like "2 vCPUs and 8GB RAM" to the closest matching instance type. You can verify the available instance types and their specs on your cluster:

[source,bash,role=execute]
----
oc get VirtualMachineClusterInstancetype -o custom-columns="NAME:.metadata.name,CPU:.spec.cpu.guest,MEMORY:.spec.memory.guest" | grep u1
----

----
u1.2xlarge    8    32Gi
u1.2xmedium   2    4Gi
u1.4xlarge    16   64Gi
u1.8xlarge    32   128Gi
u1.large      2    8Gi
u1.medium     1    4Gi
u1.micro      1    1Gi
u1.nano       1    512Mi
u1.small      1    2Gi
u1.xlarge     4    16Gi
----

== OS Selection

The agent supports multiple Linux operating systems. When you specify an OS in natural language (e.g., "create a Fedora VM"), the agent selects the appropriate container disk image from the golden images in the `openshift-virtualization-os-images` namespace.

You can check which OS images are available on your cluster:

[source,bash,role=execute]
----
oc get datasource -n openshift-virtualization-os-images
----

----
NAME              AGE

centos-stream9    17d
fedora            17d
rhel10            17d
rhel7             17d
rhel8             17d
rhel9             17d
----

[cols="2,2",options="header"]
|===
| Operating System | Default Username
| Fedora (default) | `fedora`
| RHEL | `cloud-user`
| CentOS Stream | `centos`
|===

NOTE: The container disk images are resolved automatically by the `vm_create` tool from DataSource references. You can also provide a custom container disk image URL if needed, but the agent will warn you to verify the image source is trusted before proceeding.

== Storage Configuration

The `vm-create` skill supports the following storage options:

[cols="1,3",options="header"]
|===
| Disk Size | Description
| 30Gi (default) | Suitable for most workloads
| 50Gi | Moderate storage needs
| 100Gi | Large requirements — databases, application data
| Custom | Specify any size (e.g., "200Gi")
|===

The agent automatically:

* Queries the cluster for available StorageClasses
* Identifies the default StorageClass (annotated with `storageclass.kubernetes.io/is-default-class`)
* Analyzes the StorageClass capabilities (volume binding mode, CSI provisioner)
* Creates a DataVolume-backed PersistentVolumeClaim with the appropriate size

You can check your cluster's available StorageClasses:

[source,bash,role=execute]
----
oc get sc
----

----
NAME               PROVISIONER   RECLAIMPOLICY   VOLUMEBINDINGMODE      ALLOWVOLUMEEXPANSION   AGE
lvms-vg1 (default) topolvm.io    Delete          WaitForFirstConsumer   true                   17d
----

== Network Setup

By default, the agent creates VMs connected to the cluster's default pod network, which provides basic connectivity for most use cases.

If your cluster has secondary network attachment definitions configured, you can request additional network interfaces in your natural language prompt (e.g., "create a VM with a secondary network on vlan-100"). The agent passes the network configuration to the `vm_create` tool.

== Human-in-the-Loop Confirmation

Every VM creation includes an explicit confirmation step. The agent will never create cluster resources without your approval:

. The agent gathers all parameters — from your prompt, interactive questions, and cluster defaults.
. A configuration table is displayed showing every value that will be used.
. You choose one of:
** **yes** — proceed with creation
** **no** — cancel the operation
** **modify** — change specific parameters (e.g., "modify: change storage to 50Gi")
. After modification, the updated table is shown and you confirm again.

This safety model prevents unintended resource consumption and gives you full visibility into what will be created.

== Automatic Error Diagnosis

If the VM enters an error state after creation, the agent automatically diagnoses the issue using its built-in troubleshooting knowledge base.

=== Scheduling Errors (ErrorUnschedulable)

The three most common causes are:

. **Node taints** (most common) — Worker nodes have taints (e.g., `virtualization=true:NoSchedule`) that the VM does not tolerate. The agent can add the required tolerations to the VM spec and restart it.

. **Insufficient resources** — The cluster does not have enough CPU, memory, or storage on any node for the requested VM size. The agent suggests reducing the VM size or checking node resource availability.

. **Node selector mismatch** — The VM requires specific node labels that do not exist in the cluster. The agent can remove the selector or suggest adding labels to nodes.

=== Storage Errors (ErrorDataVolumeNotReady)

Common storage issues include:

* **DataVolume still provisioning** — Large disk imports can take several minutes. The agent monitors progress and reports when provisioning completes.
* **StorageClass not found** — The requested StorageClass does not exist. The agent lists available options and offers to recreate the VM with a valid StorageClass.
* **Insufficient storage quota** — The namespace quota does not allow the requested disk size. The agent reports current quota usage and suggests alternatives.

In all cases, the agent presents a clear diagnosis with recommended solutions and waits for your decision before taking any corrective action.

== Accessing Your VM

After the VM is created and running, you can connect using several methods:

[cols="1,3",options="header"]
|===
| Method | Command
| Serial console | `virtctl console <vm-name> -n <namespace>`
| VNC console | OpenShift Console -> Virtualization -> VirtualMachines -> select VM -> Console tab
| SSH | Get the VM IP from the VirtualMachineInstance, then `ssh <user>@<ip>`
| Port forwarding | `virtctl port-forward vmi/<vm-name> -n <namespace> 8080:80`
|===

To find the VM's IP address:

[source,bash,role=execute]
----
oc get vmi fedora-vm -n my-vms -o jsonpath='{.status.interfaces[0].ipAddress}'
----

----
10.128.0.102
----

=== Default Credentials

Each OS image ships with a default user account. Console access is available immediately; for SSH, you need to set a password or inject an SSH key:

[source,bash,role=execute]
----
# Connect via serial console and set a password
virtctl console fedora-vm -n my-vms
# Once logged in:
sudo passwd fedora
----

[cols="2,1",options="header"]
|===
| Operating System | Default Username
| Fedora | `fedora`
| RHEL | `cloud-user`
| CentOS Stream | `centos`
|===

== Cleanup

To remove a VM and its associated storage:

[source,bash,role=execute]
----
oc delete vm <vm-name> -n <namespace>
----

Deleting the VirtualMachine resource also deletes the associated DataVolume and PersistentVolumeClaim. To remove all VMs created in this tutorial:

[source,bash,role=execute]
----
oc delete vm fedora-vm -n my-vms
oc delete vm fedora-dev -n development
----

You can also use natural language: type "Delete VM fedora-vm in namespace my-vms" and the agent will use the `vm-delete` skill, which includes a name-confirmation safety check before deletion.
